01What's actually good (don't touch)

Any honest overengineering review has to start by admitting where restraint was shown, so the cuts below read as surgery and not vandalism.

  • The bracket is honestly narrow. Bracket.seed_eight hardcodes 8 warriors and the QF/SF/F structure and raises on anything else. That is textbook YAGNI — no generic n-player tournament engine nobody asked for.
  • Judge eval technique is legit. Per-judge 50/50 position-bias label swap + un-swap, self-judge exclusion (a judge model can't score a match it's fighting in), and majority-with-DISCARD are real evaluation hygiene, not gold-plating.
  • Dependencies are boring and correct. openai (gateway is OpenAI-compatible), instructor for structured verdicts, textual, click, pydantic. Nothing exotic; no framework smuggled in for a few lines.
  • Tests target the pure logic (damage map, ELO, bracket seeding/propagation, majority vote, config) and skip the TUI/network — the correct things to test.

03Overengineering, ranked

FINDING 01 · biggest complexity-to-value gap

The typed event bus exists for consumers that don't

10 pydantic event classes · asyncio.Queue · dispatcher task · JSON replay codec

The engine never calls the TUI; it pushes typed events into a queue that a dispatcher drains (events.py, tui/app.py). The stated justification is a "future Unity/web renderer." That renderer is in the README's "Stretch work (not in MVP)" section. For one consumer, a direct call or a plain callback does the same job. Two of the ten event types — TournamentStarted and ResponseComplete — are emitted (driver.py:50, battle.py:77) but consumed nowhere: absent from the live _handle_event dispatch and from the replay codec's 8-of-10 mapping (app.py:176). The deserializer even raises ValueError on any unmapped kind (app.py:188) — but that guard is itself dead code: its only caller wraps it in except Exception: continue (app.py:111). So the "replay format" credited below silently can't round-trip 2 of the 10 events its own producers emit, and the check meant to catch that can never fire — dead on arrival, twice over.

Redeemed byThe demo command genuinely replays recorded events from a fixture — that reuse justifies roughly half the layer. But the shipped demo_tournament.json only decodes because it was hand-trimmed to the 8 mappable kinds (zero tournament_started/response_complete rows); the codec is a partial subset, not a general one. DoKeep the queue (replay needs it); delete the 2 zero-consumer events; stop marketing "future renderers" as architecture.

FINDING 02 · illusory precision

Bradley-Terry MLE to rank 8 players from 7 games

69 LOC · 100 iterations · log-mean anchoring · convergence tol 1e-6

Single elimination produces 7 outcomes on a sparse 8-node graph where most pairs never meet and no pair meets twice. Fitting a maximum-likelihood paired-comparison model to that yields a number with four-figure confidence and near-zero information — the champion's "ELO" is dominated by who they happened to be bracketed against. The machinery (precomputed n_matrix, geometric-mean normalization, iterate-to-tolerance) is genuinely nice code solving a problem the data can't support.

DoRank by wins / rounds-survived (≈10 lines) and label it honestly, or keep BT-MLE only if you run a round-robin. Don't sell 7 games as a rating system.

FINDING 03 · speculative interop surface — with a reuse root cause underneath it

Battlebench-compat token fields nothing writes — because a fake tokenizer stands in for the real API

tokens_a_in · tokens_b_in · tokens_*_out ← battle.py:75 max(1, len(full)//4)

BattleRecord is documented "byte-identical" to orq-battlebench so the log is drop-in for router training — a real, stated goal. The input-token fields are never populated, and the output-token fields are filled by tokens_out = max(1, len(full) // 4) (battle.py:75, whose own comment reads "We don't have usage tokens from streaming; approximate by char/4") — a hand-rolled fake tokenizer. That premise is false. The gateway is OpenAI-compatible: passing stream_options={"include_usage": True} to the streaming create() call (orq_gateway.py:54) makes the API emit a terminal chunk carrying exact prompt_tokens / completion_tokens — and the code to read resp.usage already exists, stranded in the dead complete() path (orq_gateway.py:84). So the "compatibility" gap and the char/4 estimator are one bug, not two: real usage is a single API flag away and the reader is already written.

DoSet stream_options={"include_usage": True}, read the terminal usage chunk, delete the char/4 line — then the battlebench token fields are measured, not aspirational.

FINDING 04 · reuse / reinvention — the axis the first draft under-graded

A canonical type re-spelled by hand, and a solver the ecosystem already ships

Two un-aliased verdict Literals (7 sites) · hand-rolled Bradley-Terry vs choix · comment rot

  • Two parallel verdict unions, only one aliased — and it's still re-spelled. The panel outcome Literal["A", "B", "TIE", "DISCARD"] is aliased once as Verdict (judges/panel.py:20) yet re-spelled by hand at arena/damage.py:27, events.py:72, and data/schemas.py:34 instead of imported. Meanwhile the judge choice Literal["A", "B", "TIE"] — a distinct 3-value type, since one judge cannot DISCARD — has no alias at all and is spelled out 4× (judges/schemas.py:34/44/47, events.py:64). Change either verdict set and up to 6 sites drift silently. The fix is two exported aliases, not one: collapsing both into Verdict would let a judge emit DISCARD and break the invariant the split protects.
  • Bradley-Terry is a solved, packaged problem. Finding 02's 69-LOC MLE reinvents choix.ilsr_pairwise (or ~15 lines on numpy/scipy). The file's own header — "Pure Python, no numpy/scipy" (elo.py:3) — sells the reinvention as a virtue. If a paired-comparison MLE is warranted at all (Finding 02 argues it isn't, at 7 games), it's warranted via the maintained solver, not a bespoke iterate-to-tolerance loop.
  • A docstring advertises code that isn't there. orcs/roster.py:1 promises "Warrior specification + default roster", but the file defines only WarriorSpec — no roster exists. Comment rot on line 1.
DoExport and import the one Verdict alias; adopt choix only if a round-robin ever justifies MLE (else drop it per Finding 02); fix the line-1 docstring.
Borderline, left alone: instructor for a two-field verdict is slightly heavy vs a JSON response_format, but it buys retries + validation for free — a fair trade. The double-KO / HP-tie resolution in battle.py is arbitrary-but-documented; acceptable for a game.

Aggregate: B− (held down by the reuse downgrade this revision). This is not a mess — it's a tidy codebase that mistook a weekend game for a platform. The fix isn't a rewrite; it's deletion and reuse: cut the dead ~65, collapse the ELO to a win-count, import the one Verdict type instead of four copies, swap the char/4 estimator for real stream_options usage, and stop treating "future renderer" as a load-bearing requirement. The same product then ships at ~1,650 lines with an A− on every axis above.

06Recommended cut list

  1. Delete the 8 dead items in §04 — mechanical, no behavior change, ~65 LOC and one file gone.
  2. Collapse elo.py to wins/rounds-survived ranking (~57 LOC saved) unless a round-robin mode is on the roadmap — then keep MLE and add CIs back.
  3. Trim the event layer: drop the two zero-consumer events; keep the queue (replay uses it) but reframe it as "TUI + replay," not "future renderers."
  4. Fill the battlebench token fields for real: add stream_options={"include_usage": True}, read the terminal usage chunk (the reader already exists in the dead complete() path), and delete the char/4 estimator — one change closes both §03 and half of §04.
  5. Alias the two verdict unions: export Verdict from judges/panel.py and import it at the 3 re-spelled panel sites (damage.py, events.py, data/schemas.py); add a separate JudgeChoice alias for the 3-value judge Literal spelled 4× in judges/schemas.py/events.py. Keep them distinct — a judge can't DISCARD. Fix the roster.py line-1 docstring while you're there.
  6. Leave the bracket, judge panel, and battle engine alone. They're right-sized.

Net: a leaner ~1,650-line codebase that does exactly what it does today, with the scaffolding for an imagined future taken down until that future actually files a ticket.
