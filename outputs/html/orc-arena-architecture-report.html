The code is clean, typed, documented, and tested. That is not the problem. The problem is that a 7-fight toy tournament is wearing the exoskeleton of a distributed system — a typed event bus, a JSON replay format, and a maximum-likelihood rating estimator — to serve a future (Unity/web renderers, router-training interop) that the README itself files under "not in MVP." The overengineering is real but concentrated: it lives in ~13% of the lines (the ~180 reducible + ~65 dead below, of ~1,909) and can be excised with near-zero behavior change. That was Part 1's verdict, and it stands. Part 2 (§07–§11) goes further: evaluatorq 1.3 now ships the exact machinery judges/ hand-rolls — a position-bias-corrected pairwise jury — but stronger. So the measurement layer moves into the library, the single-elim bracket gives way to a round-robin that makes the ELO honest, and this repo keeps only what nothing else provides: the show, the gateway routing, and the Bradley-Terry leaderboard. Target: ~1,600 lines that render, route, or measure — nothing else.

01What's actually good (don't touch)

Any honest overengineering review has to start by admitting where restraint was shown, so the cuts below read as surgery and not vandalism.

  • The bracket is honestly narrow. Bracket.seed_eight hardcodes 8 warriors and the QF/SF/F structure and raises on anything else. That is textbook YAGNI — no generic n-player tournament engine nobody asked for.
  • Judge eval technique is legit. Per-judge 50/50 position-bias label swap + un-swap, self-judge exclusion (a judge model can't score a match it's fighting in), and majority-with-DISCARD are real evaluation hygiene, not gold-plating.
  • Dependencies are boring and correct. openai (gateway is OpenAI-compatible), instructor for structured verdicts, textual, click, pydantic. Nothing exotic; no framework smuggled in for a few lines.
  • Tests target the pure logic (damage map, ELO, bracket seeding/propagation, majority vote, config) and skip the TUI/network — the correct things to test.

03Overengineering, ranked

FINDING 01 · biggest complexity-to-value gap

The typed event bus exists for consumers that don't

10 pydantic event classes · asyncio.Queue · dispatcher task · JSON replay codec

The engine never calls the TUI; it pushes typed events into a queue that a dispatcher drains (events.py, tui/app.py). The stated justification is a "future Unity/web renderer." That renderer is in the README's "Stretch work (not in MVP)" section. For one consumer, a direct call or a plain callback does the same job. Two of the ten event types — TournamentStarted and ResponseComplete — are emitted (driver.py:50, battle.py:77) but consumed nowhere: absent from the live _handle_event dispatch and from the replay codec's 8-of-10 mapping (app.py:176). The deserializer even raises ValueError on any unmapped kind (app.py:188) — but that guard is itself dead code: its only caller wraps it in except Exception: continue (app.py:111). So the "replay format" credited below silently can't round-trip 2 of the 10 events its own producers emit, and the check meant to catch that can never fire — dead on arrival, twice over.

Redeemed byThe demo command genuinely replays recorded events from a fixture — that reuse justifies roughly half the layer. But the shipped demo_tournament.json only decodes because it was hand-trimmed to the 8 mappable kinds (zero tournament_started/response_complete rows); the codec is a partial subset, not a general one. DoKeep the queue (replay needs it); delete the 2 zero-consumer events; stop marketing "future renderers" as architecture.

FINDING 02 · illusory precision

Bradley-Terry MLE to rank 8 players from 7 games

69 LOC · 100 iterations · log-mean anchoring · convergence tol 1e-6

Single elimination produces 7 outcomes on a sparse 8-node graph where most pairs never meet and no pair meets twice. Fitting a maximum-likelihood paired-comparison model to that yields a number with four-figure confidence and near-zero information — the champion's "ELO" is dominated by who they happened to be bracketed against. The machinery (precomputed n_matrix, geometric-mean normalization, iterate-to-tolerance) is genuinely nice code solving a problem the data can't support.

DoRank by wins / rounds-survived (≈10 lines) and label it honestly, or keep BT-MLE only if you run a round-robin. Don't sell 7 games as a rating system. Part 2resolved by more data, not less code The round-robin (§09) is now the plan: 28 matches and up to ~140 judged rounds feed the solver, ties finally flow in as 0.5/0.5 — BT-MLE becomes defensible and stays. The win-count fallback above is superseded.

FINDING 03 · speculative interop surface — with a reuse root cause underneath it

Battlebench-compat token fields nothing writes — because a fake tokenizer stands in for the real API

tokens_a_in · tokens_b_in · tokens_*_out ← battle.py:75 max(1, len(full)//4)

BattleRecord is documented "byte-identical" to orq-battlebench so the log is drop-in for router training — a real, stated goal. The input-token fields are never populated, and the output-token fields are filled by tokens_out = max(1, len(full) // 4) (battle.py:75, whose own comment reads "We don't have usage tokens from streaming; approximate by char/4") — a hand-rolled fake tokenizer. That premise is false. The gateway is OpenAI-compatible: passing stream_options={"include_usage": True} to the streaming create() call (orq_gateway.py:54) makes the API emit a terminal chunk carrying exact prompt_tokens / completion_tokens — and the code to read resp.usage already exists, stranded in the dead complete() path (orq_gateway.py:84). So the "compatibility" gap and the char/4 estimator are one bug, not two: real usage is a single API flag away and the reader is already written.

DoSet stream_options={"include_usage": True}, read the terminal usage chunk, delete the char/4 line — then the battlebench token fields are measured, not aspirational.

FINDING 04 · reuse / reinvention — the axis the first draft under-graded

A canonical type re-spelled by hand, and a solver the ecosystem already ships

Two un-aliased verdict Literals (7 sites) · hand-rolled Bradley-Terry vs choix · comment rot

  • Two parallel verdict unions, only one aliased — and it's still re-spelled. The panel outcome Literal["A", "B", "TIE", "DISCARD"] is aliased once as Verdict (judges/panel.py:20) yet re-spelled by hand at arena/damage.py:27, events.py:72, and data/schemas.py:34 instead of imported. Meanwhile the judge choice Literal["A", "B", "TIE"] — a distinct 3-value type, since one judge cannot DISCARD — has no alias at all and is spelled out 4× (judges/schemas.py:34/44/47, events.py:64). Change either verdict set and up to 6 sites drift silently. The fix is two exported aliases, not one: collapsing both into Verdict would let a judge emit DISCARD and break the invariant the split protects.
  • Bradley-Terry is a solved, packaged problem. Finding 02's 69-LOC MLE reinvents choix.ilsr_pairwise (or ~15 lines on numpy/scipy). The file's own header — "Pure Python, no numpy/scipy" (elo.py:3) — sells the reinvention as a virtue. If a paired-comparison MLE is warranted at all (Finding 02 argues it isn't, at 7 games), it's warranted via the maintained solver, not a bespoke iterate-to-tolerance loop.
  • A docstring advertises code that isn't there. orcs/roster.py:1 promises "Warrior specification + default roster", but the file defines only WarriorSpec — no roster exists. Comment rot on line 1.
DoExport and import the one Verdict alias; adopt choix only if a round-robin ever justifies MLE (else drop it per Finding 02); fix the line-1 docstring. Part 2mostly moot judges/ is deleted wholesale (§09), taking both hand-spelled Literals with it — evaluatorq's PairwiseComparison/PairwiseVote become the canonical verdict types. Only the arena-side damage map keeps a small Literal, and the BT solver stays hand-rolled-but-tested rather than importing choix (one fewer dep beats 69 borrowed lines). The docstring fix stands.
Borderline, left alone: instructor for a two-field verdict is slightly heavy vs a JSON response_format, but it buys retries + validation for free — a fair trade. The double-KO / HP-tie resolution in battle.py is arbitrary-but-documented; acceptable for a game.

Part 2 · The evaluatorq refactor

07The brief, and what evaluatorq 1.3 actually ships

The repo's end state, per the owner: an arena benchmark that produces an ELO ranking over a pool of models, wearing a TUI good enough to front a Show-HN post, and existing to promote two products — the orq.ai router/gateway (every token flows through it) and evaluatorq (every verdict comes from it). Tournament theatrics beyond that are scope creep. That hands Part 2 a one-line test for every module: render the show, route the tokens, or demonstrate the eval — or go.

evaluatorq (~52k LOC, 1.3.0-unreleased) is Orq's evaluation framework: an async data × jobs × evaluators loop with parallelism, Rich progress tables, optional Orq-platform result upload, and OpenTelemetry tracing. Two recent additions are what make this migration possible at all:

llm_jury_pairwise — the arena's judging layer, done properly

RES-760 / ADR-24 added preference judging on top of the shared jury core. The methodology is strictly stronger than judges/panel.py:

  • Both-orderings position-bias control. Every judge sees (A,B) and (B,A), run concurrently. A judge that answers the same in both frames casts that vote; a judge that flips abstains and is recorded as a flip (PairwiseVote.flipped). The arena's 50/50 single-ordering coin toss merely dilutes position bias across rounds — it can never detect a biased judge, let alone discount one.
  • Panel resilience as parameters. replacement_judges promotes stand-ins (run in both orderings) when a primary errors; min_successful_judges turns a too-degraded panel into inconclusive instead of a fake verdict; repetitions for stochastic judges.
  • A real cross-comparison report. build_report(comparisons) rolls up A/B win rates, tie and inconclusive rates, mean inter-judge agreement, and per-judge lean / flip-rate / tie-rate. The arena computes none of this today.
  • Judge token accounting (TokenUsage summed across orderings and replacements) — the very thing Finding 03 caught the arena faking with len(text)//4.
from evaluatorq import llm_jury_pairwise, build_report

jury = llm_jury_pairwise(
    judges=["anthropic/claude-haiku-4-5", "google/gemini-2.5-flash", "openai/gpt-4o-mini"],
    criteria="Accuracy, helpfulness, clarity, relevance to the prompt.",
    replacement_judges=["mistral/mistral-small"],  # refills an eroded panel
    min_successful_judges=2,                       # jury-of-one → 'inconclusive', never a verdict
    client=gateway.client,                         # the arena's existing orq-router AsyncOpenAI
)
comparison = await jury.compare(question=prompt, response_a=resp_a, response_b=resp_b)
# comparison.winner: 'A' | 'B' | 'tie' | 'inconclusive' · per-judge votes[].flipped · token_usage

report = build_report(all_comparisons)  # A/B win rates · mean agreement · per-judge flip rate

Judge traffic routes through Orq by default: evaluatorq's client resolver honors ORQ_API_KEY / ORQ_BASE_URL, and client= accepts the arena's existing gateway AsyncOpenAI as-is. A panel is just list[str] of router model ids — JudgeSpec, the judge prompt builder, and the instructor dependency all evaporate.

Pin note: pairwise lives on the RES-760 branch and 1.3.0 is unreleased — depend on a git ref (or a local path dep) until it lands on PyPI. Also 1.3.0 makes openai and loguru core dependencies; the arena already ships openai.

llm_jury + the core loop — the offline half

llm_jury(...) builds a panel evaluator (categorical / numeric / boolean verdicts, aggregators, tie-breaks, pass/fail) for evaluatorq(evaluators=[...]), and the core loop's inference=False mode scores pre-recorded responses without regenerating them. That is exactly what battles.jsonl is — which makes "re-judge last night's tournament with a different panel, offline, zero generation cost" a two-line demo of the second product (§10, PR 4).

08Finding 05 — the shipped panel is subtly miscalibrated

Part 1 graded judges/panel.py "the actual IP" on methodology. A harder look at the shipped configuration turns up the reason "replace" beats "keep" — beyond mere deduplication.

FINDING 05 · new in part 2 · config × code interaction

Two of the three judges moonlight as warriors — and the damage math can't count

panel of 3 · self-judge exclusion · damage.py:35–37 — n_agree == n_total, where n_total = len(verdicts) or 1

Self-judge exclusion (battle.py:186) is correct hygiene — but the default orc_arena.yaml makes it bite: judges Flash (gemini-2.5-flash) and 4o-mini (gpt-4o-mini) are also seeded warriors. In the default bracket, quarterfinal Snot the Quick (gpt-4o-mini) vs Azog Deepmind excludes judge 4o-mini, and Morgoth Quadskull vs Gruk Flashfist (gemini-2.5-flash) excludes judge Flash — both fights are decided by a panel of two. compute_damage defines unanimity as "everyone left agrees" (n_agree == n_total), so every 2-0 round deals the 30-damage "unanimous" hit that the YAML explicitly reserves for 3-0 (damage_unanimous: 30  # 3-0 verdict) — and a two-judge panel can never deal the 15-damage majority hit at all: 2-0 reads as unanimous, 1-1 is DISCARD. Should Snot and Gruk both reach the final, the title fight is judged by Haiku alone — and n_total = len(verdicts) or 1 makes every decisive round of it an automatic "unanimous" 30. A judge that errors is silently dropped on top (panel.py:62 — except Exception: return None), eroding panels further with no floor.

In evaluatorqThese are parameters, not patches: min_successful_judges=2 turns a jury-of-one into inconclusive; replacement_judges refills an eroded panel (a neutral fourth model also cures the moonlighting); flips are measured per judge instead of averaged away. Unanimity for the damage map is then derived from comparison.votes, guarded by actual panel size. DoDon't fix panel.py — delete it (§09–§10), set the two parameters, and add one neutral replacement judge to the config so a real fight never falls below two reconciled votes.

Why the ELO gets honest, not just cheaper

Today the solver sees 7 match outcomes, and a drawn match is impossible — an HP tie is awarded to the higher seed (battle.py:250, comment: "seed advantage"), so the tie handling in build_wins_matrix is dead-in-practice code. The refocused arena feeds the solver per-round panel verdicts from a full round-robin: C(8,2) = 28 matches × up to 5 rounds ≈ 140 reconciled comparisons, ties entering as the 0.5/0.5 the code always supported. Finding 02's "illusory precision" dissolves — same solver, defensible sample. The 8-warrior constraint dies with the bracket: any pool size works, which is what "benchmark a pool of models" requires. HP, KO, and the round cap stay exactly as they are — as the show — but the ranking no longer depends on them.

Cost, stated plainly: full round-robin (28 vs 7 matches) with both-orderings judging (6 vs 3 judge calls per round) is roughly 8× today's judge volume (~840 calls vs ~105) and 4× the warrior-generation volume. The judges are deliberately cheap models, and every knob exists: swap=False halves judging, fewer prompts per match, smaller pools. Ship the honest default; expose the knobs in orc_arena.yaml.

Line math: −196 (judges/) − 100 (bracket + stub) − 65 (dead, ~10 overlap) + ~40 (scheduler + adapter + config glue) ⇒ ≈ −310 net, landing at ~1,600 src LOC — a 16% smaller repo whose judging got strictly stronger. Tests: test_panel_tally.py and test_bracket.py retire with their subjects; one small adapter test takes their place.

11The two-product story — and what the TUI keeps

After the four PRs, every part of the demo maps to a product pitch:

  • orq router/gateway — eight models from six vendors fight through one OpenAI-compatible client and one API key (api.orq.ai/v2/router). The streaming fight screen is literally gateway throughput as spectacle.
  • evaluatorq — every verdict on screen is a consistency-gated, position-bias-corrected panel decision; the leaderboard's per-judge stats and the rejudge command are the library's pairwise + jury + replay features, demonstrated on data the audience just watched being made.
  • The benchmark — battles.jsonl accumulates real cross-vendor comparison data with honest token counts; the ELO table is the reusable output.

The TUI loses nothing and gains drama. Streaming panes, HP bars, and the KO banner stay untouched.
