01What chennai is

A Conductor worktree on the personal repo (arianpasquali/orc-arena, branch arena-modes-brainstorm) with its own 42-commit history from Initial commit — not a fork of this repo's code, but its conceptual ancestor. It answers "what if the arena did everything": run (single-elim) + arcade (double-elim with losers bracket) + simulate (round-robin/Swiss) + debate (Lincoln-Douglas protocol with a two-phase judge committee), a mode picker, a CRT-neon theme, an audio player, per-model LLM post-mortems, a cost engine, a battle browser, and a fixture record/replay system. Judging is the hand-rolled 50/50-swap panel this repo just deleted; there is no evaluatorq anywhere.

Amusing convergence: chennai's single extra fix on the shared sidequests master — renaming JudgeCard._render because it shadows Textual's internal — is the same bug this repo hit and fixed independently yesterday.

03The harvest, ranked by value-per-line

HARVEST 01 · port · roster_select.py (629) + providers/models_list.py · ~700 LOC adapted

Interactive roster picker over the workspace-enabled catalog

Run orc-arena run with no --config → a picker opens: live-search input, provider filter chips, a seed-order roster panel that fills as you pick, and a live cost estimate updating with every selection. Behind it sits chennai's best research artifact, models_list.py: GET /v2/router/models is the workspace-enabled subset (models disabled in the Model Garden simply don't appear), intersected with GET /v2/models for authoritative type == "chat", cached 24h, with a refresh-models CLI command. This repo's roster pain — a broken preview model and think-by-default surprises discovered mid-smoke — is exactly what picking from the enabled catalog mitigates.

WhyKills YAML-editing friction; showcases the router's catalog as a product surface; "benchmark your pool" becomes literal. AdaptDrop the {2,4,8,16,32} bracket-size gate (round-robin takes any ≥2); default-theme styling; picked roster carries the uniform-OFF reasoning policy with a 🧠 toggle per pick later.

HARVEST 02 · deferred (owner, 2026-07-12) · analysis/cost.py (390) + prices.yaml · dollar math postponed; the judges-vs-warriors split ships in tokens (exact) instead

Cost engine: pre-run estimate, post-run actuals, judges-vs-warriors split

prices.yaml (USD per Mtok, optional cached_input, default fallback with an explicit "this is a guess" flag), estimate_tournament_cost for the picker/preflight, and compute_actual_cost feeding a leaderboard panel that separates what the jury cost from what the fighters cost. This is PR 5's cost-preflight item, already built — and the split is an honesty feature: with both-orderings judging, the jury bill is the methodology's price tag, in dollars.

DeferralOwner call: a hand-maintained price table is a guess that goes stale monthly — too loose for a benchmark that advertises exactness. What ships now: the judges-vs-warriors split in exact token counts (PR 5), plumbed so a future prices.yaml multiply-through lands as one small isolated PR.

HARVEST 03 · dropped (owner, 2026-07-12) · replay/recorder.py + --record flags + demo --refresh + pacing keys

Fixture recorder — good code, wrong surface

JSONL with real inter-event pacing, atomic .partial finalize, --record flags, demo --refresh. Owner call: too much machinery for one consumer. The demo command itself stays exactly as shipped — ~30 lines plus one committed fixture, the only zero-key path and the CTA's trigger. But recording exists solely to regenerate that fixture when the schema changes (~quarterly), which is a dev task: it becomes scripts/record_fixture.py (~50 lines, key required, run rarely), not CLI surface. Pacing keys dropped with it — the fixture's curated delays are fine.

HARVEST 04 · port · battle_browser.py (343 adapted) · leaderboard key B

Battle browser — step through every judged round after the run

Prompt, both responses, every judge's verdict and reasoning, damage and HP deltas — one round per page, arrow-key navigation, straight off battles.jsonl. For a benchmark, this is the trust feature: spot-check the jury before you believe the leaderboard. Adapting to schema v2 makes it better than the original: flip/abstain badges and reasoning-token counts come along free.

HARVEST 05 · port · tui/theme.py (61) + widget touch-ups

The CRT-neon arcade theme

A deliberate, documented palette ("near-black with a whisper of violet, never pure #000") with markup helpers and HP color ramps. This repo's TUI runs on stock Textual dark — functional, anonymous. For a Show-HN artifact the visual identity is the thumbnail. Side-identity semantics (A green / B orange) survive inside the neon palette (P1 magenta / P2 cyan is their equivalent — pick one pair, keep it everywhere).

HARVEST 06 · adapt · analysis/model_analysis.py (317) + screen (329) → rebuilt thinner

"Why did I lose" — per-model LLM post-mortems

After a run, one background task per model digests its battles — prompts, its own responses, the judges' free-text reasoning — through an analyzer LLM into a structured summary, cached in analysis.jsonl, rendered from the leaderboard. Genuinely novel and very Show-HN ("Grak wins on completeness, bleeds points to verbosity"). Rebuild, don't port: chennai uses instructor (which this repo just removed); the same feature is ~150 lines on the existing client's structured output, fed by PairwiseVote.explanation — richer input than v1 had.

HARVEST 07 · adapt · analysis/judge_stats.py (299) → ~80 LOC of it

Fleiss' κ and Cohen's κ — chance-corrected jury agreement

Textbook implementations with Landis-Koch labels ("substantial agreement"), pure Python, cited (Fleiss 1971, Cohen 1960, Zheng et al. 2023). evaluatorq's build_report gives raw modal agreement and flip rates; κ corrects for chance — "mean agreement 61%" and "κ = 0.18, slight agreement" tell very different stories, and the second is the one a statistician will ask for. Take the κ math, skip their position-bias section (superseded by evaluatorq's both-orders flips). Lands in the leaderboard jury panel and run.json.

HARVEST 08 · adapt · SwissScheduler (~70 LOC) from round_robin.py

Swiss pairing for large pools — same benchmark, scaled

Score-group pairing with rematch avoidance, and a real statistical argument in the docstring: BT converges faster when strong plays strong; random pairing needs ~3× the matches for the same ranking confidence. Not a mode — an auto-switch by pool size (≤8 → full round-robin, >8 → Swiss), which is what "ELO over a pool of models" needs the day someone points it at 20 models. Wire it to per-round outcomes (their version feeds match-level, no ties — the flaw this repo already fixed).

HARVEST 09 · adapt · match concurrency (semaphore + standings lock) from round_robin.py

Concurrent matches — for headless runs only

Chennai runs up to simulation.concurrency matches in parallel; events interleave but carry match_id. The TUI stays one-fight-at-a-time (it's the show), but PR 5's --headless CI mode wants exactly this: a 20-model Swiss run overnight is wall-clock hostile sequentially. ~30 lines on the existing driver, gated behind headless.

HARVEST 10 · adapt · cta_modal.py (78)

Post-demo call-to-action

Demo ends → modal: "run it live with your own pool — export ORQ_API_KEY" with play-live/quit keys. The repo exists to promote two products; this is the one moment a delighted demo viewer is asked to become a user. Tiny, unobtrusive, worth it.

One-line verdict: chennai answered "what could the arena be" with everything; this repo answers "what should it be" — and ten of chennai's answers were right. Take the picker, the prices, the recorder, the browser, the κ, the post-mortems, the theme, the Swiss, the concurrency, and the CTA. Leave the modes in the museum.
